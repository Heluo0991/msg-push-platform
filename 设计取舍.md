# 设计取舍记录

## SPSC 无锁队列 × N vs SPMC 有锁池

### 选择：SPMC 有锁池（ThreadPool 内置 `std::queue + mutex`）

### 两条路

| | N 个 SPSC 无锁 | 1 个 SPMC 有锁 |
|------|------|------|
| 数据结构 | Main → N 个 SPSC 队列，Worker 各绑一个 | Main → ThreadPool::tasks_，Worker 抢活 |
| 锁 | 零锁，atomic | 一把 mutex |
| 持锁时间 | 无 | ~几十纳秒（`push/emplace`） |
| 负载均衡 | 轮询不均，某 Worker 吃满、某 Worker 饿 | 自动均衡，空闲线程立刻抢 |
| 线程池语义 | 非池——线程被孵化后锁定在永久循环里，永不归还 | 真池——submit 一个任务，抢到的人干完回来再抢 |
| 析构停机 | 线程锁在 while(true) 里，ThreadPool::stop_ 看不见 | 线程在 cv_.wait 里，stop_→notify_all→join 标准流程 |
| 复杂度 | N 个队列数组 + 轮询 + stop_flag + 信号处理 | ThreadPool 原始 API，不加一行代码 |

### 为什么 SPSC 无锁没选这个场景

SPSC 无锁队列在本项目里没有消失——`lockfree_queue.h` 保留在 Day 1 产物中，作为技术展示。Day 5 没有用它的原因是：

1. **ThreadPool 本身已是 SPMC 模型**——`cv_.wait + mutex + queue` 完成了单生产（Main submit）多消费（Worker pop）的完整语义。
2. **业务耗时远大于锁开销**——JSON 解析 + DB 查询都是毫秒级，几十纳秒的锁争持可忽略。
3. **负载均衡更优**——N 个 Worker 抢一个任务队列，自然分布。轮询 N 个独立队列反而可能导致某 Worker 堆积、某 Worker 空闲。
4. **线程池语义正确**——`submit` 投递的 lambda 执行完就返回，Worker 回到池子里等下一个任务。ThreadPool 的 `stop_` 机制正常工作，析构停机干净。

### 放弃 SPSC 的具体代价

- Main 线程 `submit` 时多持一次纳秒级锁
- N 个 Worker 抢同一把锁——概率极低的排队（4 线程 × 微秒级业务耗时）

### SPSC 的正确位置

在这个项目里，SPSC 的适用场景是 **Main → Worker 的"同一个固定 channel"**——比如每个 Worker 自己绑定一个专属管道。但当 Worker 共享同一个池化层时，强行拆 N 个管道相当于为每个水管开一个泵——泵多了，水管利用率反而不均。

## 其他设计决策

### 网络模型：epoll ET + 非阻塞 fd

> 为什么 ET 不是 LT：ET 一次通知 + drain 清空，LT 可能导致同一 fd 反复通知浪费 epoll_wait。listen_fd 必须同时设非阻塞——`accept4` 的 `SOCK_NONBLOCK` 只影响新 fd，listen_fd 自己也要 `socket(SOCK_NONBLOCK)`，否则 ET 循环 accept 时第二次调用会阻塞卡死。

### 消息协议：`\n` 分隔符 + JSON 体

> 为什么不用长度前缀：聊天消息是纯文本 JSON，`\n` 分隔简单直观，`nc` 直接测试。长度前缀对二进制更友好，但 JSON 场景下可读性优先。

### 环形缓冲区：固定 4096 字节数组

> 为什么不是 `std::string` 动态扩容：固定大小防止恶意客户端持续发送不带 `\n` 的数据撑爆内存。4096 字节足够蹲几十条聊天消息。两指针 + 位掩码取模，单线程用不需要 atomic。

### 单头文件第三方依赖：nlohmann/json.hpp

> 为什么允许这一个第三方库：手写 JSON 解析不值得——逃逸、嵌套、Unicode 等边界条件工作量巨大，且对面试官展示无积极意义。一个头文件零链接、零构建系统依赖，不违反"零第三方 C++ 库"原则。

### 数据库：SQLite

> 为什么不是 MySQL：云服务器仅 1.5GB 内存，MySQL 装不下。SQLite 零部署、零进程，一个 `messages.db` 文件，本地和远程环境统一。`prepare + bind` 防注入，单连接串行模式天然线程安全。

### 构建系统：Makefile

> 为什么不是 CMake：单项目 ~12 个源文件，Makefile 30 行够用。CMake 同样多行反而复杂。

### 消息类型：variant + visit + overloaded

> 为什么不是虚函数继承链：variant 编译期多态零虚表开销，6 种类型在编译期生成跳转表，运行时一次 index → switch 完成。`overloaded` 模板把 6 个 lambda 拼成重载集，比 if-else 链干净、比虚函数快。

### 连接生命周期：shared_ptr + weak_ptr

> 为什么不是裸指针：Main 持有 shared_ptr，Worker 持有 weak_ptr。Worker 干活前 `lock()` 检测连接是否存活，防止主线程关闭连接后 Worker 访问已析构对象。

### 对象池：placement new + union 空闲链表

> 为什么不让 OS 每次 new/delete：256 个 Connection 槽位预分配，避免频繁系统调用。union 复用空闲链表指针和对象内存——同一块内存在不同时刻充当不同角色。LIFO 栈式归还，最近释放的槽位还在 CPU 缓存里热着。

### Broker 设计：namespace + 自由函数

> 为什么不是 Broker 类：业务逻辑函数只依赖参数引用（connections、db_store），不需要成员变量。自由函数传引用更灵活，不需要给每个 handler 包一层 this 捕获。

### SQL：prepare + bind 永远不拼字符串

> 为什么：手拼 SQL 字符串是注入漏洞的根源。`?` 占位符 + `sqlite3_bind_text` 让 SQLite 内部处理引号和转义。用户输入 `alice'); DROP TABLE users; --` 也不会产生任何副作用。

### 注册/登录合并：一条 LoginMsg

> 为什么分开：客户端不需要知道"账户是否存在"。服务器先 `verify_login`，失败就 `register_user`——新用户第一次登录自然被注册。PRIMARY KEY 冲突是最后防线，不会覆盖已有密码。

### 日志：fprintf(stderr, ...)

> 为什么不是 iostream：`fprintf` 无虚函数调用，无 `sync_with_stdio` 同步开销。`stderr` 独立于 stdout，即使重定向也打印到终端。C 风格在底层 I/O 场景反而是最轻的选择。

### 状态门卫：dispatch 开头检查 State::LOGIN

> 为什么不在 protocol 层过滤：protocol 只管 JSON 格式，不管业务规则。client 可能发任意 JSON 格式的消息，服务器自己防住"未登录先广播"的逻辑错误。

### 信号处理：epoll_wait 1s 超时 + atomic stop flag

> 为什么不是 eventfd 唤醒：epoll_wait 1s 超时轻量，轮询检查 `running_`。eventfd 需要在 epoll 中额外注册一个 fd、修改 reactor 事件分发逻辑——单进程服务器不需要这么复杂的停机机制。

### 缓存行隔离：alignas(64)

> 为什么两根手指分开：Main 写 `write_idx_`、Worker 读 `read_idx_` 逻辑上互不干扰，但若在同一个 64 字节缓存行内，CPU 的 MESI 协议会使两个线程互相踢对方的缓存。加 56 字节空洞推入不同缓存行消除 false sharing。

### ET 竞态修复：accept 后立即 drain + handle_read

> 问题：客户端在服务器 accept 拿到 fd 之后、subscribe 注册到 epoll 之前的瞬间，数据已经到达内核缓冲区。ET 只看"注册后从空到有"的边沿——数据注册前就在了，对 epoll 来说这个 fd 从未经过"空→有"的边沿变化，永远收不到通知。管道测试（`echo | nc`）中连接和发送几乎同时发生，极容易触发。

> 解法：accept 之后立刻手动 `drain` 一次——不管数据到了没，先把内核缓冲区清空拉到自己的 ring_buffer 里。然后立刻手动调一次 `handle_read`。send 没来？ring_buffer 空的，try_pop break，无副作用。send 来了？数据在 ring_buffer 里等着，handle_read 拿到就提交给 Worker 处理。

> 核心思路：靠 epoll 的通知不够快——我自己主动先看一眼。drain 做搬运，handle_read 做立刻处理。注册前到的数据不会漏，注册后到的数据照常走 epoll 通知。两个时间窗口全盖上。

### 单元测试：纯 assert + 手动 nc

> 为什么不用 gtest：9 个 .cpp 测试文件每次 `make test` 一键跑完，零框架依赖，零学习成本。Reactor 级别用 nc 手动验证——网络集成测试不自动化比写 mock 更可靠。

### Reply 存放 variant 而非预序列化 JSON 字符串

> 为什么：Worker 处理完后直接往 MPSC 塞 variant，序列化留给主线程。Worker 永远不碰 `send/send_line`，网络 I/O 完全在主线程内完成——不会跨线程调 socket。

### 连接池与 epoll 实例同生命周期

> 为什么不用全局单例：`connections_` 和 `epfd_` 都属于 Reactor 实例，同生同死。析构时 `ObjectPool` 晚于 `connections_` 声明（反序析构保证安全），shared_ptr 自定义删除器自动归还对象池。

### 登录竞态保护：`compare_exchange_strong` 而非 `mutex`

> 问题：同一连接短时间内发两条 LoginMsg，两个 Worker 同时 load 到 `state_ == LOGIN`，都通过 guard，都写 `set_state` 和 `set_username`。`string` 并发写是未定义行为。

> 为什么不是 `mutex` 锁整个 guard：一条连接的登录只发生一次，极罕见的竞争窗口。mutex 锁一大片代码只为一次罕见事件——走内核态开销，杀鸡用牛刀。

> 为什么选 `compare_exchange_strong`：`state_` 是 `atomic<State>`。`compare_exchange_strong(expected, CHAT)` 是一条硬件指令——"如果 state_ 还是 LOGIN，改成 CHAT 并返回 true；否则不改，返回 false"。两个 Worker 同时到达时，CPU 在缓存行层面只让一个通过。失败的那个直接返回 `"frequent_login"` error，不碰 username_。

> 额外收益：`try_claim_login()` 只一行代码，无锁、无异常路径。`set_state(CHAT)` 也省了——comparing exchange 已经同时完成了检查和修改。一条 `LOCK CMPXCHG` 指令解决了 guard + writer 两个问题。

### 群组成员存储：纯内存，不持久化

> 问题：`groups_` 哈希表（群名 → 成员 fd 集合）存在内存里，服务器重启群组全部丢失。为什么不用数据库建 `group_members` 表？

> 理由：这是消息推送中台，不是 QQ。群成员在线才有意义——离线的人即使记录在群里，下次登录也收不到这段期间的历史消息（没有消息历史表）。群组是会话级的，和在线连接同生命周期。服务器重启 → 所有连接断开 → 群组自然消亡，下次在线重新建群即可。

> 面试追问应对："群组状态的生命周期绑定在连接上。持久化群组需要额外的 `group_members` 表 + `group_messages` 历史表，那是社交网络的消息存储，超出了推送中台的定位。当前设计下群组就是一个广播组——在线的人收到，离线的人收不到。真正的离线消息由 private_msg + `offline_msgs` 表负责。"
